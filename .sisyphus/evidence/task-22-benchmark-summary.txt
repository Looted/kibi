# Kibi Performance Benchmarks - v0 Baseline

## Overview
Baseline performance measurements for kibi v0 using mitata benchmark library.
These are reference measurements before optimization work.

## Sync Performance

### Results
- **10 files**: ~1.38s avg (1.36s - 1.41s)
- **100 files**: ~5.93s avg (5.87s - 6.03s)  
- **Incremental (1/100 changed)**: ~11.70s avg (11.59s - 11.77s)

### Analysis
- Sync time scales linearly: ~59ms per file for 100 files
- Incremental sync is 2x slower than full sync (needs investigation)
- v0 baseline is 6x slower than target (< 1s for 100 files)

### Performance Bottlenecks (to investigate)
1. Prolog process startup overhead
2. RDF write operations (semweb library)
3. File I/O and entity extraction
4. Incremental sync doing full resync (no delta detection)

## Query Performance

### Results
- **100 entities (all)**: ~6.75s avg (6.67s - 7.00s)
- **100 entities (by ID)**: ~6.84s avg (6.67s - 7.23s)

### Analysis
- Query time dominated by workspace setup (init + sync)
- Actual query execution likely < 100ms (hidden in setup overhead)
- Benchmark design issue: measuring setup + query, not pure query time
- By-ID query has same performance as full scan (setup dominates)

### Needs Refinement
- Separate setup from query measurement
- Add persistent workspace benchmarks (query-only)
- Measure pure Prolog query time without process startup

## MCP Latency

MCP benchmark not completed due to time constraints. Each MCP call includes:
1. Stdio JSON-RPC parsing
2. Prolog process management
3. Tool execution
4. JSON-RPC response formatting

Estimated latency > 5s per call (dominated by workspace setup in current implementation).

## Key Learnings

### Benchmark Design
1. **Setup vs Execution**: Current benchmarks measure full workflow (init + sync + query) rather than isolated operations
2. **Warm vs Cold**: All benchmarks measure cold start (new workspace each time)
3. **Realistic Data**: Generated markdown files with frontmatter simulate real usage

### Performance Characteristics
1. **Sync is CPU-bound**: Each file requires parsing, entity extraction, Prolog upsert
2. **Prolog overhead**: Starting/stopping Prolog process is expensive
3. **RDF persistence**: Semweb's snapshot + journal pattern has write overhead
4. **No caching**: v0 has zero caching - every operation is cold

### Optimization Opportunities (Future)
1. **Persistent Prolog process**: Reuse across operations instead of restart
2. **Delta sync**: Only process changed files in incremental sync
3. **Batch operations**: Bulk upsert instead of one-by-one
4. **Query result caching**: Cache frequently-accessed entities
5. **Lazy loading**: Load RDF graphs on-demand instead of upfront

## Targets vs Reality

| Operation | Target | v0 Baseline | Gap |
|-----------|--------|-------------|-----|
| sync 100 files | < 1s | ~6s | 6x slower |
| query 1000 entities | < 100ms | ~7s* | 70x slower* |
| MCP tool call | < 50ms | ~5s+ | 100x+ slower |

*Note: Query benchmark includes setup overhead, not pure query time

## Next Steps

1. **Refine query benchmark**: Separate setup from measurement
2. **Add warm cache benchmarks**: Measure with persistent workspace
3. **Profile hot paths**: Identify actual bottlenecks (Prolog? I/O? Parsing?)
4. **Optimize incrementally**: Target 2x improvements per iteration
5. **CI integration**: Track performance regressions automatically

## Conclusion

v0 performance is far from targets, but this is expected. Benchmarks establish baseline for future optimization work. Main insight: **setup overhead dominates** - need persistent processes and better caching.
