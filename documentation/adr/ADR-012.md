# ADR-012: Branch Truth, Publish Policy, and Diagnostics Enforcement

## Context

Kibi's per-branch knowledge base (KB) architecture enables isolated, queryable project memory for each git branch. However, inconsistencies in branch selection and KB attachment have led to two critical issues:

1. **Stale branch attachment**: KBs are not always updated correctly on checkout, merge, or pull, leading to agents or users operating on outdated or incorrect branch data.
2. **Weak enforcement of first-class scenario/test entities**: The system has not strictly enforced that scenarios and tests are always present and up-to-date for each branch, undermining traceability and coverage guarantees.

Current codebase inconsistencies:
- `init-helpers.ts` defaults to `develop`
- `sync.ts` defaults to `main`
- `server.ts` (MCP) defaults to `develop`

This ADR establishes a unified, explicit policy for branch truth, override precedence, publish atomicity, and diagnostics, locking down the rules for all future KB operations.

## Decision

### 1. Branch Selection and Precedence

Branch selection for KB operations MUST follow this explicit precedence order:

1. **`KIBI_BRANCH` environment variable**: If set, this value is authoritative and overrides all other sources. Used for automation, CI, and explicit override scenarios.
2. **Active git branch**: If `KIBI_BRANCH` is unset, the currently checked-out git branch (from `git rev-parse --abbrev-ref HEAD`) is used.
3. **Diagnostic failure**: If neither is available (e.g., detached HEAD, unborn branch, or non-git workspace), the operation MUST fail with an explicit diagnostic. No silent fallback to any default branch is permitted.

### 2. Default Branch Resolver for New Branch KBs

When creating a KB for a new branch, the source KB is copied from the resolved default branch. The default branch is determined in this order:
1. `.kb/config.json` `defaultBranch` (if set)
2. `origin/HEAD` (if available)
3. `main` (fallback only if neither of the above is set)

This ensures compatibility with repositories that use a non-`main` default branch and allows explicit configuration.

### 3. Override and Marker Behavior

- Any `.kb/active` or similar marker file is **advisory only** and MUST NEVER override the above precedence. It may be used for UI hints or debugging, but not for authoritative branch selection.
- No silent fallback to `main`, `develop`, or any other branch is allowed except as the explicit fallback in the default branch resolver above.
- All override or fallback behavior must be surfaced as explicit diagnostics.

### 4. Supported Lifecycle Events

The following lifecycle events MUST be supported for branch-aware KB operations:
- **Checkout**: On branch checkout, the KB for the new branch is created (if missing) by copying from the resolved default branch (see above), then synced.
- **Merge**: On merge, the KB is synced to reflect the merged state.
- **Pull**: On pull, the KB is synced to reflect upstream changes.

Detached HEAD, unborn branch, or non-git workspace:
- All KB operations MUST fail with a clear diagnostic. No silent fallback or implicit branch creation.

### 5. Atomic Publish Unit

The atomic unit of publish is the **full branch snapshot directory** under `.kb/branches/<branch>`. All KB state, audit logs, and derived artifacts for a branch are published, versioned, and validated as a single unit.

### 6. Diagnostics Taxonomy

Diagnostics for branch and publish operations MUST include:
- **Branch selection errors**: (e.g., detached HEAD, unknown branch, missing KB)
- **Override conflicts**: (e.g., both `KIBI_BRANCH` and git branch set but differ)
- **Lifecycle event failures**: (e.g., failed KB copy, sync, or validation)
- **Non-goal diagnostics**: (see below)

### 7. Explicit Non-Goals (First Wave)

The following are **explicitly deferred** and NOT part of this ADR:
- Resumable jobs for long-running KB operations
- Rollback or undo framework for failed publishes
- Automatic migration or repair of malformed documentation
- Any redesign of the KB entity/relationship model
- Generic MCP simplification or unrelated roadmap items

## Consequences

- **Positive**: Ensures all KB operations are branch-true, explicit, and diagnosable. Prevents silent data drift and stale context.
- **Positive**: Locks down atomicity and diagnostics for publish, enabling robust automation and auditability.
- **Positive**: Default branch resolver supports non-main setups and explicit configuration, eliminating hardcoded assumptions.
- **Negative**: Users in detached HEAD or non-git workspaces must resolve context before proceeding; no fallback.
- **Negative**: Deferred features (rollback, resumable jobs) may be requested in future ADRs.
