## Context

Recent releases of kibi packages worked correctly in development but failed when installed from npm. Issues included:
- Missing runtime assets (Prolog files, schema files) not included in the `files` array
- Path resolution assumptions that worked in monorepo but failed in installed packages
- Bin entries with incorrect permissions or missing shebangs
- Integration tests timeout due to concurrent Prolog process contention

We need automated E2E tests that verify packages work AS CONSUMERS WOULD USE THEM, not just in the development environment.

## Decision

Implement E2E testing using **temp directories + per-test npm prefixes** with the following architecture:

### Isolation Strategy
- Each test scenario runs in complete isolation:
  - Fresh temp directory (`mkdtemp`)
  - Fresh git repository (`git init -b develop`)
  - Fresh npm prefix (`npm_config_prefix=<tmp>/npm-prefix`)
  - Fresh npm cache (`npm_config_cache=<tmp>/npm-cache`)
  - Fresh `$HOME` directory (prevents dotfile pollution)
  - Isolated `PATH` including only the test prefix bin

### Package Installation
- Tests install from tarballs created by `npm pack` (not workspace symlinks)
- This catches missing files, incorrect bundling, and packaging errors
- Example: `npm install -g --prefix /tmp/test-prefix ./kibi-cli-0.1.4.tgz`

### Prolog Process Handling
- **NO shared Prolog pools** across test scenarios
- Each `kibi` invocation spawns its own Prolog child process
- Parallelism achieved via **CI job matrix** (VM-level isolation), not in-process concurrency
- Serial execution within each job (`--test-concurrency=1`)

### Test Structure
```
documentation/tests/e2e/packed/
├── helpers.mjs                    # Test harness utilities
├── cli-workflows.test.mjs         # CLI install + basic commands
├── hooks.test.mjs                 # Git hook execution
└── mcp.test.mjs                   # MCP server stdio operations
```

## Rationale

### Why temp dirs over Docker?
- **Faster execution**: No container startup overhead
- **Simpler maintenance**: No Docker image to maintain
- **Sufficient isolation**: Per-test npm prefixes + fresh git repos provide adequate isolation
- **Easy local development**: Developers can run tests without Docker

### Why tarballs over workspace paths?
- Catches packaging errors (missing files in `files` array)
- Tests the exact artifact consumers receive
- Validates `npm pack` behavior matches publishing
- Prevents "works in dev, fails in production" regressions

### Why serial execution?
- Eliminates Prolog process contention (root cause of test timeouts)
- Simpler to debug (no race conditions)
- CI job matrix provides parallelization without resource conflicts

## Consequences

### Positive
- Catches packaging errors before releases
- Prevents Prolog contention timeouts
- Can run locally without Docker
- Validates real consumer experience

### Negative
- Slightly slower than parallel in-process tests
- Requires careful environment isolation
- More CI jobs (matrix) increases total CI time

## Implementation

### Phase 1: Core Harness
Create test utilities and basic scenarios:
- `helpers.mjs`: packing, installation, git repo creation, process execution
- `cli-workflows.test.mjs`: install, doctor, init, sync, check
- `hooks.test.mjs`: hook installation and execution
- `mcp.test.mjs`: stdio JSON-RPC lifecycle

### Phase 2: CI Integration (Optional)
Add to `.github/workflows/ci.yml`:
```yaml
e2e-packed:
  strategy:
    matrix:
      suite: [cli-workflows, hooks, mcp]
  steps:
    - run: node --test --test-concurrency=1 documentation/tests/e2e/packed/${{ matrix.suite }}.test.mjs
```

### Phase 3: Release Gate (Optional)
Add to publish workflow to run E2E before promoting to `latest`.
