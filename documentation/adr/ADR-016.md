## Context

The OpenCode Kibi Plugin v1 requires a clear separation of host glue and Kibi logic, as well as a robust fallback policy for prompt injection and sync.

## Decision

- All OpenCode host glue code must reside in `packages/opencode` as a dedicated host package.
- All Kibi logic (sync, MCP, KB operations) must remain in the Kibi CLI and MCP server packages, not duplicated in the plugin host.
- The plugin must act as a thin bridge: it shells out to `kibi sync` and directs agents to existing MCP tools, never re-implementing Kibi logic.
- Hook contract lock (validated by `packages/opencode/tests/hook-contract.test.ts` against local installed types):
  - Primary hook: `experimental.chat.system.transform` for injected Kibi guidance that must reach model prompt text.
  - Fallback hook: `chat.params` for option-level enrichment only (`temperature`, `topP`, `topK`, `options`) and never for prompt text.
- Documentation requirement: package README (Task 10) must restate the same primary/fallback hook policy.
- This policy is locked for v1 and must not be relaxed without a new ADR.

## Rationale

- Keeps plugin host minimal and maintainable.
- Ensures all Kibi logic is single-sourced and testable.
- Prevents prompt injection assumptions that conflict with the installed OpenCode contract.
- Allows for robust fallback if OpenCode contract changes.

## Verification

- Plugin code must reference `REQ-opencode-kibi-plugin-v1` for traceability.
- Contract tests must prove hook behavior from installed types, not public docs alone.
- Tests must verify correct package placement and fallback behavior.
