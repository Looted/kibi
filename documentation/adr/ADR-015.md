## Context

Issue #58 reported a high-severity consistency bug in `kibi-mcp@0.2.1`: `kb_upsert` and `kb_delete` could report success while subsequent reads in the same MCP session still returned stale data. The inconsistency was most visible when a read query was executed before a mutation and repeated after the mutation.

Investigation showed that `PrologProcess` query caching was invalidated after successful mutations only in one-shot mode, but not in interactive mode (the mode used by long-lived Node MCP server sessions).

## Decision

On every successful non-cacheable query in interactive mode, invalidate the query cache.

- Keep read-query caching for performance.
- Treat all mutation/query goals classified as non-cacheable as cache invalidation boundaries.
- Align cache behavior between one-shot and interactive execution paths.

## Rationale

- Interactive MCP sessions must provide read-after-write consistency.
- Cache invalidation on successful mutations is simpler and safer than selective cache key invalidation.
- The one-shot path already followed this contract; this decision removes mode-specific behavior drift.

## Implications

- Cached read results are dropped after successful writes in interactive sessions.
- Mutation visibility is consistent across `kb_query` by `id` and by `type` listing in a single MCP process.
- A persistent-session regression test is required to prevent regressions.

## Verification Strategy

- Add an E2E regression test that keeps one MCP server process alive and validates:
  1. create -> read -> update -> read shows updated fields,
  2. type-list query includes the created entity,
  3. delete -> read returns not found.
