## Context

Kibi is distributed as a monorepo with multiple packages that have different release targets and consumption models. There was ambiguity about which packages are published to npm versus bundled or published elsewhere, leading to documentation drift.

## Decision

**Decision**: Establish a clear package matrix where `kibi-core`, `kibi-cli`, and `kibi-mcp` are release-managed npm packages with independent versioning, while `kibi-vscode` is published exclusively to the VS Code Marketplace.

### Package Matrix

| Package | Directory | npm Name | Published To | Current Version | Dependencies |
|---------|-----------|----------|--------------|-----------------|--------------|
| kibi-core | `packages/core/` | `kibi-core` | npm | 0.1.6 | None |
| kibi-cli | `packages/cli/` | `kibi-cli` | npm | 0.2.0 | `kibi-core: ^0.1.6` |
| kibi-mcp | `packages/mcp/` | `kibi-mcp` | npm | 0.2.1 | `kibi-core: ^0.1.6`, `kibi-cli: ^0.2.0` |
| kibi-vscode | `packages/vscode/` | N/A | VS Code Marketplace | N/A | N/A |

### Versioning Model

- **Independent versioning**: Each npm package maintains its own version in `package.json`
- **Changesets-style selective publishing**: The publish workflow (`.github/workflows/publish.yml` and `scripts/publish-selective.sh`) only publishes packages whose local version does not yet exist on the npm registry
- **Semver ranges**: Packages declare dependencies using caret (`^`) ranges to allow compatible updates
- **No workspace protocol**: Published packages use concrete version ranges instead of `workspace:*` to ensure they work when installed outside the monorepo

### Release Process

1. **CI/CD Gate**: The `publish.yml` workflow includes a `release-gate` job that:
   - Packs all three npm packages
   - Verifies tarball contents include necessary files
   - Performs strict install smoke test from packed tarballs
   - Validates MCP server startup

2. **Selective Publishing**: Only packages with versions not present on npm are published:
   ```bash
   # Workflow checks each package
   npm view ${pkg_name}@${version} version
   # If not found, mark for publishing
   ```

3. **Provenance**: All npm publishes include `--provenance` for supply chain security

## Rationale

- **kibi-core as independent package**: While kibi-core contains only Prolog files (no compiled JS), publishing it separately allows:
  - Proper dependency declaration in kibi-cli and kibi-mcp
  - Independent versioning of the core graph logic
  - Potential external consumption by other tools
  
- **Three npm packages vs bundled**: Keeping core separate (rather than bundling into cli/mcp) maintains:
  - Clear separation of concerns
  - Smaller package sizes for cli/mcp
  - Ability to patch core independently

- **vscode excluded from npm**: The VS Code extension has different distribution mechanics and marketplace requirements

## Consequences

### Positive
- Clear contract: README, workflows, scripts, and package.json all agree on the package matrix
- Selective publishing reduces unnecessary releases
- Tarball verification catches packaging errors before publish
- Independent versioning allows targeted updates

### Negative
- Must coordinate version bumps across three packages when core changes
- Release process requires all three packages to build and pack even if only one changes
